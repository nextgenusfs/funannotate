# Trinity Rust Integration Summary

## Overview
Integration of Rust-optimized Trinity utility binaries into funannotate workflow. These high-performance replacements handle computationally intensive SAM processing and coverage analysis tasks used during Trinity's genome-guided assembly.

## Rust Trinity Tools

### 1. **sam_to_read_coords**
- **Source**: `rust_bio_utils/src/bin/sam_to_read_coords.rs`
- **Replaces**: `util/support_scripts/SAM_to_frag_coords.pl`
- **Purpose**: Parse SAM files and extract read coordinate information
- **Output Format**: Tab-separated coordinates (scaffold, read_name, pair_side, read_start, read_end)
- **Input**: SAM files (or stdin)
- **Performance**: Rust implementation with optimized parsing and I/O buffering

### 2. **extract_reads_per_partition**
- **Source**: `rust_bio_utils/src/bin/extract_reads_per_partition.rs`
- **Replaces**: `util/support_scripts/extract_reads_per_partition.pl`
- **Purpose**: Extract reads from SAM files based on partition GFF regions
- **Input**: 
  - Partitions GFF file (coverage regions)
  - Coordinate-sorted SAM file
- **Output**: Reads organized into partition directories
- **Options**:
  - `--partitions_gff <file>`: GFF file defining partitions
  - `--coord_sorted_SAM <file>`: Coordinate-sorted SAM file
  - `--parts_per_directory <int>`: Partitions per directory (default: 100)
  - `--min_reads_per_partition <int>`: Minimum reads to process (default: 10)
- **Performance**: Parallel processing with HashMap-based partition tracking

### 3. **fragment_coverage_writer**
- **Source**: `rust_bio_utils/src/bin/fragment_coverage_writer.rs`
- **Replaces**: `util/support_scripts/fragment_coverage_writer.pl`
- **Purpose**: Generate WIG format coverage files from fragment coordinates
- **Input**: Fragment coordinates file (scaffold, frag_name, lend, rend)
- **Output**: WIG format coverage data per scaffold
- **Format**: 
  - Input must be sorted by scaffold, then position
  - Output: variableStep WIG format
- **Performance**: Single-pass streaming with accumulated coverage tracking

### 4. **define_coverage_partitions**
- **Source**: `rust_bio_utils/src/bin/define_coverage_partitions.rs`
- **Replaces**: `util/support_scripts/define_coverage_partitions.pl`
- **Purpose**: Define partition regions based on coverage thresholds
- **Input**: 
  - WIG coverage file with strand-specific data
  - Minimum coverage threshold
  - Strand identifier (+/-)
- **Output**: GFF partition regions
- **Format**:
  - Input: WIG with "variableStep chrom=<scaffold>" headers
  - Output: GFF lines with partition regions where coverage >= threshold
- **Performance**: Single-pass streaming with region merging

### 5. **libtrinity_bio.so** (shared library)
- **Purpose**: Provides core bio-utilities as a shared library
- **Used by**: Other Rust tools and potentially external C/C++ code
- **Installation**: `/opt/conda/lib/libtrinity_bio.so`

## Integration Details

### Build Process
The Trinity Rust tools are built during pixi environment activation via `scripts/pixi_install_trinity_rust.sh`:

```bash
cd ../trinityrnaseq/rust_bio_utils
cargo build --release
# Binaries installed to CONDA_PREFIX/bin
```

### Installation Path
- **Binary Installation**: `${CONDA_PREFIX}/bin/`
  - `sam_to_read_coords`
  - `extract_reads_per_partition`
  - `fragment_coverage_writer`
  - `define_coverage_partitions`
- **Library Installation**: `${CONDA_PREFIX}/lib/libtrinity_bio.so`

### Trinity Workflow Integration

These utilities are used internally by Trinity during:

1. **Genome-Guided Assembly (Trinity --genome_guided_bam)**
   - BAM alignment to genome is generated by HISAT2
   - Rust utilities partition coverage into assembly regions
   - Trinity processes each region in parallel
   - Coverage data is generated in WIG format

2. **Process Flow**
   ```
   SAM file (from HISAT2)
          ↓
   sam_to_read_coords → read coordinates
          ↓
   fragment_coverage_writer → WIG coverage file
          ↓
   define_coverage_partitions → GFF partition regions
          ↓
   extract_reads_per_partition → reads per partition directory
          ↓
   Trinity (per-partition assembly)
   ```

- Provides: Perl-based versions as fallback

**Rust tools**: Supplement conda Trinity installation with performance-optimized versions

## Performance Improvements

Rust implementations provide:
- **Faster I/O**: Buffered reading/writing for large SAM/coverage files
- **Memory efficiency**: Streaming processing (no full file loads)
- **Parallel processing**: Rayon-based parallelization where applicable
- **Reduced overhead**: No Perl interpreter startup cost

## Dependencies

- `rust >=1.80` (for cargo build)
- `cargo` (Rust package manager)
- Standard build tools (gcc, make)

## Testing
